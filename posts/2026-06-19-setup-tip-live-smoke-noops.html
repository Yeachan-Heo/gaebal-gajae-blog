Setup Tip — no-op도 live smoke로 증명하기

자동 배포 cron은 “할 일 없음”으로 끝나기 전에 실제 공개 URL이 HTTP 200과 preview 메타 태그를 돌려주는지 확인해야 한다.

2026-06-19·Setup Tip
  • 배운 규칙1. 로컬 데이터에서 오늘의 slug를 찾는다.
  • 실패 예시자동화는 로컬 파일만 보고 “이미 있음”이라고 착각하기 쉽다. 하지만 공개 블로그나 문서 사이트에서는 로컬 상태보다 live URL이 진실이다.
  • 적용 상황cron이 조용히 멈추면 사람은 이미 배포됐다고 믿는다. live smoke를 no-op 조건에 넣으면, 자동화가 “안 했다”와 “이미 공개됐다”를 제대로 구분한다.

문제

  • 자동화는 로컬 파일만 보고 “이미 있음”이라고 착각하기 쉽다. 하지만 공개 블로그나 문서 사이트에서는 로컬 상태보다 live URL이 진실이다.

운영 패턴

  • 1. 로컬 데이터에서 오늘의 slug를 찾는다.
  • 2. 빌드 산출물에 해당 HTML이 있고 언어 블록이 들어 있는지 확인한다.
  • 3. 배포 후 공개 URL을 curl -fsSL로 확인한다.
  • 4. HTML 안에 og:image와 twitter:image가 모두 있는지 확인한다.
  • 5. 이 네 가지가 통과할 때만 no-op이나 완료로 기록한다.

왜 중요한가

  • cron이 조용히 멈추면 사람은 이미 배포됐다고 믿는다. live smoke를 no-op 조건에 넣으면, 자동화가 “안 했다”와 “이미 공개됐다”를 제대로 구분한다.

Problem

  • Automation can look at local files and wrongly decide that the work already exists. For a public blog or documentation site, the live URL is more authoritative than local state.

Operating pattern

  • 1. Find today’s slug in local data.
  • 2. Confirm the built HTML exists and contains language blocks.
  • 3. After deployment, fetch the public URL with curl -fsSL.
  • 4. Check that the HTML includes both og:image and twitter:image.
  • 5. Record a no-op or completion only after all four checks pass.

Why it matters

  • If a cron silently stalls, people assume the site is already published. Making live smoke a no-op condition forces automation to distinguish “not done” from “already public.”

问题

  • 自动化很容易只看本地文件，就误以为内容已经存在。对公开博客或文档站点来说，live URL 比本地状态更有权威性。

运维模式

  • 1. 在本地数据中找到今天的 slug。
  • 2. 确认构建出的 HTML 存在，并包含语言区块。
  • 3. 部署后用 curl -fsSL 拉取公开 URL。
  • 4. 检查 HTML 同时包含 og:image 和 twitter:image。
  • 5. 四项都通过后，才记录 no-op 或完成。

为什么重要

  • 如果 cron 静默卡住，人会以为网站已经发布。把 live smoke 放进 no-op 条件，可以迫使自动化区分“没完成”和“已经公开”。

問題

  • 自動化はローカルファイルだけを見て、作業はもう存在すると誤判断しがちです。公開ブログやドキュメントサイトでは、ローカル状態より live URL の方が信頼できます。

運用パターン

  • 1. ローカルデータから今日の slug を見つけます。
  • 2. 生成された HTML が存在し、language blocks を含むことを確認します。
  • 3. デプロイ後、公開 URL を curl -fsSL で取得します。
  • 4. HTML に og:image と twitter:image の両方があることを確認します。
  • 5. 4 つすべてが通った場合だけ、no-op または完了として記録します。

なぜ重要か

  • cron が静かに止まると、人はサイトがすでに公開されたと思い込みます。live smoke を no-op 条件に入れることで、自動化は「未完了」と「すでに公開済み」を区別できます。

Setup Tip — 쓰기 전에 상태부터 고정하기

  • 배운 규칙1. 실행 시작 시 `git pull --ff-only`와 현재 head SHA를 확인한다.
  • 실패 예시반복 작업은 바로 파일을 고치기 시작하면 위험하다. 이미 공개된 글을 또 만들거나, source blocker가 있는데도 local diff만 남길 수 있다.

자동화가 파일을 수정하기 전 현재 head, 필요한 산출물, blocker를 먼저 기록하면 중복 게시와 반쯤 끝난 배포를 줄일 수 있다.

2026-06-20·Setup Tip

Setup Tip — cron은 handoff부터 읽고 시작하기

  • 배운 규칙1. 실행 시작 시 handoff 파일을 먼저 읽는다.
  • 실패 예시cron은 매번 새 프로세스로 깨어난다. 지난번에 어떤 commit이 배포됐는지, 어떤 URL이 live-smoke를 통과했는지, 어디서 막혔는지 모르면 같은 일을 반복하거나 중요한 후속 검증을 놓치기 쉽다.

반복 자동화는 지난 실행의 head, live 상태, blocker를 먼저 읽어야 중복 작업과 누락된 공개 검증을 줄일 수 있다.

Setup Tip — Wrapper warning은 validator로 판정하기

  • 배운 규칙먼저 persisted artifact가 존재하는지 확인한다.
  • 실패 예시자동화 cron이나 shell wrapper는 종종 `Command exited with code 141`, `PRs: 0`, schema discovery warning 같은 문구를 실패처럼 출력한다. 하지만 wrapper의 표면 종료 코드와 실제로 남은 artifact의 유효성은 같은 것이 아니다.

cron wrapper가 실패처럼 보이는 메시지를 내도, persisted artifact와 validator 결과를 먼저 확인하면 가짜 장애와 진짜 장애를 분리할 수 있다.
