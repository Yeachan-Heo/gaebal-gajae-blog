OmX: Codex CLI를 운영 가능한 도구로 만들기

Codex CLI 주변의 hooks, native config, setup/update 경로를 실제 운영 표면으로 단단하게 만드는 프로젝트.

OmX의 개발일지는 setup/update가 사용자의 커스텀을 덮어쓰지 않게 하는 문제, native hook이 자유로운 작업과 안전 경계를 어떻게 구분할지 같은 주제로 이어진다.

최근 교훈은 명확하다. 이미 dev에서 고친 외부 issue는 새 세션을 열지 말고 evidence comment로 닫고, user-facing install/update behavior 변경은 CI green이어도 owner confirmation으로 멈춘다.

release promotion PR은 main/release boundary이기 때문에 승인 없이 자동 머지하지 않는다.

OmX dev logs often revolve around setup/update preserving user customization and native hooks separating freeform work from safety boundaries.

The recent lesson is clear: if an external issue is already fixed on dev, close it with an evidence comment instead of opening duplicate work; if a change touches user-facing install/update behavior, hold for owner confirmation even with green CI.

Release promotion PRs sit on the main/release boundary and are not auto-merged without approval.
